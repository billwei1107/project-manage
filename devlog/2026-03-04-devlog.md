### Troubleshooting: Login Failure (Connection Refused)
- **問題描述**: 使用者反映無法登入，經查無法連線至 Backend (Port 8081)。
- **原因分析**: 先前在 `User` 實體中新增外加 `NOT NULL` 的 `isOnline` 布林欄位：`@Column(nullable = false)`，但這在現有已包含資料的 PostgreSQL 表中 (`users`) 會引發 Schema Migration (DDL) 失敗，導致 Spring Boot Context 啟動 Crash。
- **解決動作**: 修正 `User.java`，於新增的攔位加入 DDL 預設值限制：`@Column(name = "is_online", nullable = false, columnDefinition = "boolean default false")`，讓 Hibernate 得以順利完成 `ALTER TABLE`。

### Troubleshooting: Blank Employee List & Unauthorized Access
- **問題描述**: 管理員看到空白的員工列表，且非管理員有可能看到左側選單的「員工列表」項。
- **原因分析**:
  1. `HrService.java` 中的 `getAllEmployeesBrief()` 方法裡在建構回傳的 Map 時，使用了 Java 9 的 `Map.of()` 方法。由於 `Map.of()` 拒絕接受 `null` 作為 Value，而在 `user.getLastLoginAt()` 為 null（從未登入過）時便會引發 `NullPointerException` 導致 API 炸鍋回傳 500。
  2. `AdminLayout.tsx` 的路由選單中，原先把 `員工列表` 寫在全局開放區域。
- **解決動作**:
  1. 將 `Map.of()` 重構為傳統的 `new HashMap<>()`。
  2. 在 `HrController.java` 對讀取 API 加上 `@PreAuthorize("hasRole('ADMIN')")` 以確保權限。
  3. 在 `AdminLayout.tsx` 移除了全局的員工列表入口，並放在 `user?.role === 'ADMIN'` 判斷條件中。

### Troubleshooting: 404 Endpoint Not Found

- **解決動作**: 將 `/hr/employees` 改為 `/v1/hr/employees`，並修正 UI Empty Table 將 `colSpan={4}` 擴展至 `colSpan={6}` 避免排版跑位。

### Troubleshooting: Login Time Not Updating & Missing Employees
- **問題描述**: 登入後 `isOnline` 和 `lastLoginAt` 沒有更新，且員工列表只顯示一名管理員 (其他帳號不見)。
- **原因分析**:
  1. `AuthService.java` 中原本的登入流程只負責核發 JWT，並未主動實作回寫 `user.setOnline(true)` 與時間紀錄的邏輯。
  2. `HrService.java` 的 `updateHeartbeat` 原先只用 `findByEmail` 或 `findById` 收尋使用者，但認證 `auth.getName()` 提供的不見得符合。
  3. `HrService.java` 的 `getAllEmployeesBrief()` 中預設過濾了 `CLIENT` 角色的用戶，導致非管理員被錯誤隱藏。
- **解決動作**:
  1. 於 `AuthService.java` 成功認證後寫入 `isOnline=true` 與 `lastLoginAt=now()` 並呼叫 `userRepository.save()`。
  2. 將 `HrService` 尋找使用者的語法統一改為 `findByUsernameOrEmployeeIdOrEmail` 確保完全精準配對。
  3. 移除了 `getRole() != User.Role.CLIENT` 過濾條件，現在員工列表會忠實呈現所有類別的註冊系統帳號。

### Troubleshooting: Offline Status Sticking (Not Auto-Offlining)
- **問題描述**: 登出後，或者放著不管網頁關掉，使用者的線上狀態綠燈不會媳滅 (變成離線)。
- **原因分析**:
  1. Frontend 的 `useAuthStore.ts` 中的 `logout` 只有把 LocalStorage 給清空，並沒有發送請求給後端解除狀態。
  2. 後端缺乏一個檢查 Time Out 的背景排程，以清查中斷連線 (未發送 Heartbeat) 的使用者。
- **解決動作**:
  1. 於 `ErpApplication.java` 加入 `@EnableScheduling`。
  2. 在 `HrService.java` 新增 `@Scheduled(fixedRate = 180000)` 自動排程任務，找出超過 5 分鐘未寄送 Heartbeat 的 User 並標記 `isOnline = false`。
  3. 於 `AuthController` 開放一個主動的 `POST /api/v1/auth/logout` 端點讓前端可以主動通知後端使用者登出，並在 `AuthService.java` 中將狀態設為離線。

### Troubleshooting: Timezone Offset (UTC Time shown rather than Local Time)
- **問題描述**: 系統紀錄的登入時間比本地時間慢了 8 個小時 (顯示 3:00 而非 11:00)。
- **原因分析**: Spring Boot 服務器預設啟動時會抓取 OS 的時區或預設為 `UTC+0`。這會導致透過 `LocalDateTime.now()` 存入 DB 的時間戳記出現 8 小時落差，並且前端顯示時又未進行轉換，產生混淆。
- **解決動作**: 於 Spring Boot 的啟動主類別 `ErpApplication.java` 加入 `@PostConstruct` 生命週期方法，強制把整個 JVM 應用程式的預設時區指定為 `Asia/Taipei` (`TimeZone.setDefault(...)`)，確保後續所有的日誌及 `LocalDateTime.now()` 皆為台灣時間。

### Feature & Troubleshooting: Employee ID Auto-Generation & False Occupied Errors
- **問題描述**: 
  1. 新建帳號時，需要根據角色 (Role) 來決定員工編號，而不是統一的 `EMP`，且如果未指定則自動產生序列 (例如 `ADM001`, `DEV002`)。
  2. 新增帳號時如果留空沒給資訊，會一直報錯提示「用戶名或信箱已存在 (Occupied)」。
- **原因分析**: 
  1. 原本的 ID 產生機制統一寫死了 `String.format("EMP-%04d", nextId)` 無法區隔身分。
  2. 後端 `User.java` 實體標記了 `username`、`employeeId` 等皆為 `@Column(unique = true)`。當用戶連續建立多筆未指定選填欄位的帳號時，前端傳來的皆為空字串 `""`，在 PostgreSQL DB 層第一筆空字串會通過，但第二筆空字串就會觸發 `Unique Index Constraint Violation` 演變成假性佔用。
- **解決動作**: 
  1. **優化 ID 產生器**: 於 `AccountService.java` 重構了 `generateEmployeeId` 加入了 `switch(role)` 來賦予不同的前綴 (`ADM`, `DEV`, `PM`, `CLI`)。
  2. **加載最大值前綴收尋法**: 於 `UserRepository.java` 中加入了 `findTopByEmployeeIdStartingWithOrderByEmployeeIdDesc(prefix)` 方法，精準尋找出該部門最新的一組編號再累加生成新的。
  3. **空字串處理 (Nullify Empty Strings)**: 於 `AccountService.java` 的 `createUser()` 及 `updateUser()` 中，如果發現選填欄位 (`username`, `email`, `githubUsername`) 傳來的是空字串 `""`，將其全部轉換為 `null` 才存入 DB，以避免資料庫把空字串當作有意義的 Unique 重複值。
